# 从书到 Skill：知识提炼工作流

**硬规则：每一步验证不过，不准进入下一步。**

## 什么时候用

用户提供一本书的文件，想把书里的知识沉淀成可复用的 Hermes skill。

触发语：书的文件名、路径，加上「生成 skill」「沉淀成技能」「从这本书学」「提炼成 skill」等。

## 什么时候 NOT 用

- 用户只是问书的内容、要点 → 直接回答，不生成 skill
- 没有提供文件，只说书名 → 先要求提供文件（PDF/Markdown/TXT）

---

## ⚠️ CRITICAL PRINCIPLES（non-negotiable）

1. **不编造命令/API** — 书里没有的不能凭空写。书里的命令直接引用，推断的部分必须标注 `（推断）`
2. **不依赖网络搜索** — 所有内容必须来自用户提供的文件，不能去网上搜书籍摘要/书评来补
3. **长书必须分块** — 超过 2000 行的文件不能一次性读，必须先看目录再选核心章节
4. **质量验证不过就回修** — 不跳过 checklist，不让不合格的 skill 进 _drafts/

---

## 流程概览

```
0. 前置判断（书适合吗？）→ NO → 终止
1. 读取 & 分析（PDF 转文本 → 读目录 → 判断类型 → 选骨架）
2. 提炼核心（填统一入口 + 类型专属章节）
3. 生成 SKILL.md → 写入 _drafts/
4. 质量验证（跑 checklist）
5. Self-test（书中真实场景验证）
6. 交付（输出摘要 → 等确认 → 移动到正式目录）
```

## 两种模式

在 0 阶段开始时询问用户：

> 用审查模式（每步暂停确认）还是快速模式（一口气生成，最终确认）？

- **审查模式**（默认）：0 → 暂停 → 1 → 暂停 → 2 → 暂停 → 3 → 4 → 5 → 6
- **快速模式**：0 → 1 → 2 → 3 → 4 → 5 → 6（只在最终暂停）
- 每个暂停点给用户看产物 + 问「继续？」。用户说改就回上一步。

---

## 阶段 0：前置判断

### 0.1 检测文件格式

| 格式 | 处理方式 |
|------|---------|
| `.md` / `.txt` | 直接 read_file |
| `.epub` | 用 Python 解压 ZIP，提取 HTML/XHTML 文本，输出到 `/tmp/book_to_skill.txt`。脚本见 `references/epub-convert.md`。 |
| `.pdf` | `pdftotext -layout <file> /tmp/book_to_skill.txt`，然后读 txt |

**PDF 转换命令**：
```bash
pdftotext -layout "原文件路径" /tmp/book_to_skill.txt
```
如果 pdftotext 不可用 → 安装 poppler-utils：
```bash
sudo apt-get install poppler-utils   # Debian/Ubuntu
brew install poppler                  # macOS
```

### 0.2 判断是否适合生成 skill

读目录 + 前 200 行，判断书属于以下哪一类：

**A. 操作知识类**（技术书、方法论书）—— 回答三个问题：
1. 书中有**可操作的知识**吗？（命令、步骤、原则、方法、框架）
2. 这些知识能**被 agent 使用**吗？（不是纯人类体验）
3. 有一个**明确的应用场景**吗？

三个全 YES → 继续。→ 生成类型：工具手册/操作规范/实战+顾问。

**B. 思维模型类**（小说、传记、历史、哲学）—— 回答三个问题：
1. 书中有一个**鲜明的角色/作者视角**吗？（角色如何分析问题、做决策）
2. 这个视角能提炼成**可复用的思维模型**吗？（不是泛泛的「给人启发」）
3. Agent 面对类似**决策困境时能用上**吗？

三个全 YES → 继续。→ 生成类型：决策框架/响应协议。

**A 和 B 都不满足 → 终止。**

判断示例：
- 《1984》→ B 类 ✓。温斯顿在极权下的独立思考和生存策略 → 可提炼为「受限环境中的批判性思维」框架
- 《原子习惯》→ A 类 ✓。有明确的习惯养成步骤 → 操作规范型
- 《小王子》→ A 和 B 都不满足。虽有哲理但无可复用的思维模型 → 终止

### 0.3 确认模式

用 clarify 问用户：审查模式还是快速模式？

---

## 阶段 1：读取 & 分析

### 1.1 读目录

用 read_file 读转换后的文本前 200 行，定位目录部分。

如果目录不明显，读前 500 行搜索「目录」「Contents」「Chapter」等关键词。

### 1.2 选核心章节（策略因 A/B 类而异）

**A 类（操作知识类）**—— 跳读策略：

分析目录，挑出与**可操作知识**最相关的章节：
- 跳过：前言、致谢、附录、索引、参考文献
- 跳过：纯背景介绍、作者个人故事
- 选取：核心方法、步骤指南、原则论述、案例对比、checklist

选取 3-8 章，记录章节名 + 行号范围。

**B 类（思维模型类）**—— 全读策略：

小说/传记结构连贯，跳读会丢失角色思维演变的上下文。
- 按顺序读完全书所有章节（跳过纯粹的前言/后记）
- 每章读后立即提炼：角色关键决策、思维转折、核心对话
- 不在 context 里堆积原始文本

### 1.3 分块读取

**A 类**：每章单独 read_file，提炼要点后再读下一章。

**B 类**：按顺序读完全书，每章提炼后再读下一章。

每章提炼内容：
- 1-3 句核心内容/情节
- 关键概念或角色决策
- 可用的思维模型/原则（B 类）或可操作的步骤/命令（A 类）
- 书中的反例/陷阱

不要在 context 里堆积原始文本——读完一章提炼一章，只保留提炼结果。

### 1.4 判断 skill 类型

按 0.2 的分类走不同判断路径：

**A 类（操作知识类）**——按以下信号判断：

| 信号 | 类型 |
|------|------|
| 目录/正文中大量出现命令、API、配置项、参数 | **工具手册型** |
| 核心章节标题含"How to"、"Step by Step"、"流程"、"步骤" | **操作规范型** |
| 操作步骤和设计决策各占约一半，有对比有选择 | **实战+顾问型** |

**B 类（思维模型类）**——按以下信号判断：

| 信号 | 类型 |
|------|------|
| 角色/作者的思维模式、分析框架、决策逻辑 | **决策框架型** |
| 角色/作者的表达方式、沟通策略、说服方法 | **响应协议型** |

审查模式下，在此暂停——告诉用户判断的类型 + 理由，问确认。

---

## 阶段 2：提炼核心

### 2.1 生成 skill name

规则：
- 英文、小写、连字符
- 格式：`<核心主题>-<定位>`，如 `git-workflow`、`refactoring-checklist`
- 不超过 3 个单词

审查模式下让用户确认 name。

### 2.2 填写统一入口（5 种类型共用）

按以下顺序写，每节 2-5 行：

**Overview**
一句话 elevator pitch。格式：「本书核心是 X。这个 skill 帮助 agent 在 Y 场景下做 Z。」

**Core Philosophy**
书的核心思想压缩成 3-5 条。agent 遇到 skill 没覆盖的场景时，靠这个推断。每条一句话。

---

<示例>
Core Philosophy:
1. 数据系统必须考虑故障模式——网络分区不是意外，是必然
2. 一致性、可用性、分区容忍性三者最多同时满足两个
3. 批处理和流处理的本质区别不是延迟，是正确的抽象
---

**⚠️ CRITICAL PRINCIPLES**
non-negotiable 红线。和 Core Philosophy 的区别：Philosophy 是「应该这么想」，CRITICAL 是「绝对不能这么做」。最多 5 条。

---

<示例>
CRITICAL PRINCIPLES:
- 永远不要假设网络可靠——所有远程调用必须有超时+重试
- 单点故障不可接受——任何组件都可以挂，系统不能挂
- 不要用事务解决本应是工作流的问题
---

**When to Use**
- 具体场景列表（3-5 个）

**When NOT to Use**
- 明确不该用的场景（2-3 个）——防止 agent 滥用

### 2.3 填写类型专属章节

根据阶段 1 判断的类型，选择对应的骨架：

---

#### 类型 1：工具手册型

**Quick Reference**（高频操作 5-10 条）
```markdown
| 场景 | 命令 |
|------|------|
| 查看状态 | `git status` |
| 查看差异 | `git diff` |
```
命令必须来自书，不能编造。

**Tool Boundaries**
```markdown
| 任务 | 更适合的工具 | 原因 |
|------|------------|------|
```

**Pitfalls**
```markdown
| Problem | Cause | Fix |
|---------|-------|-----|
```

---

#### 类型 2：操作规范型

**Procedure**（分步流程，每步有验证）
```markdown
### Step 1: <步骤名>
- 做什么
- ✅ 验证：<如何确认这步做对了>

### Step 2: ...
```

**Quality Gates**（每阶段的通过标准）
```markdown
| 阶段 | 通过标准 | 不通过怎么办 |
|------|---------|-------------|
```

**Pitfalls**
```markdown
| Problem | Cause | Fix |
|---------|-------|-----|
```

**Verification**
自动/手动验证命令列表。

---

#### 类型 3：决策框架型

**Decision Framework**（面对 X → 用 Y 原则 → 预期 Z 结果）
```markdown
| 场景 | 应用原则 | 预期结果 |
|------|---------|---------|
```

**Mental Models**
```markdown
### <模型名>
- 是什么（一句话）
- 什么时候用
- 例子
```

**Anti-Patterns**
```markdown
| 错误做法 | 为什么错 | 正确做法 |
|---------|---------|---------|
```

**Pitfalls**
```markdown
| Problem | Cause | Fix |
|---------|-------|-----|
```

---

#### 类型 4：实战+顾问型

**Quick Reference**（高频操作 5-10 条）

**Decision Matrix**
```markdown
| 场景 | 推荐方案 | 理由 | 替代方案 |
|------|---------|------|---------|
```

**Procedure**（复杂操作的完整流程）

**Proactive Review**（agent 应主动检查的清单）
```markdown
- [ ] 检查点 1
- [ ] 检查点 2
```

**Tool Boundaries**
```markdown
| 任务 | 更适合的工具 | 原因 |
|------|------------|------|
```

**Pitfalls**
```markdown
| Problem | Cause | Fix |
|---------|-------|-----|
```

---

#### 类型 5：响应协议型

**Response Rules**（硬约束）
```markdown
- 结构：<规定输出格式，如「3 个选项，不多不少」>
- 字数：<如有上限>
- 风格：<如「一句话一个问题」>
```

**Anti-Patterns**
```markdown
| 错误做法 | 为什么错 | 正确做法 |
|---------|---------|---------|
| 过度结构化，在不该用的场景套格式 | 让回复机械僵硬 | 判断场景是否适合，不适合则降级为自由格式 |
```

**Pitfalls**
```markdown
| Problem | Cause | Fix |
|---------|-------|-----|
```

---

### 2.4 写入 _drafts/

路径：`~/.hermes/skills/_drafts/<skill-name>/SKILL.md`

用 write_file 创建。确保 _drafts/ 目录存在：
```bash
mkdir -p ~/.hermes/skills/_drafts
```

审查模式下，在此暂停——展示完整 SKILL.md 内容，问确认。

---

## 阶段 3：生成 SKILL.md

同 2.4。如果 2 阶段已经写入，此阶段只做最终检查。

---

## 阶段 4：质量验证

跑以下 checklist，每一项必须通过：

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 1 | Overview 是不是一句话能说清 skill 是干什么的？ | 读 Overview，能 10 秒内理解 → ✓ |
| 2 | CRITICAL PRINCIPLES 是 actionable 的吗？ | 检查是否有「要重视安全」这类废话 → 有则改 |
| 3 | 有没有编造的命令/API？ | 每个命令反查书中原文 → 找不到则删或标「（推断）」 |
| 4 | 有没有 Tool Boundaries？ | 1/4/5 型必须有；2/3 型不强制 |
| 5 | When NOT to Use 写了吗？ | 必须有，至少 2 条 |
| 6 | Pitfalls 表格非空？ | 至少 3 条 |
| 7 | SKILL.md 总行数是否合理？ | 50-200 行为佳；超过 300 行考虑拆分到 references/ |

任何一项不通过 → 回阶段 2 修改。

---

## 阶段 5：Self-test

从书中抽一个**真实场景**（书里描述过的问题或案例），用刚生成的 skill 回答：

1. 描述场景（1-2 句）
2. agent 用这个 skill 应该给出什么答案？
3. 这个答案是否：
   - 用到了 skill 里的原则/命令/框架？
   - 合理且可操作？
   - 不是泛泛的废话？

不合理 → 说明 skill 提炼不够 → 回阶段 2 修改。

---

<示例>
场景：用户在 main 分支上直接改代码，现在想把改动移到新分支。
期望答案：用 `git stash` + `git checkout -b <branch>` + `git stash pop` 三步。
如果 skill 里没有覆盖这个高频场景 → 补充到 Quick Reference。
---

---

## 阶段 6：交付

### 6.1 输出摘要

```markdown
## 生成完成：<skill-name>

**类型**：<类型名>
**来源**：<书名>
**核心章节**：<选取的章节列表>

**一句话概述**：<Overview 内容>

📁 草稿路径：~/.hermes/skills/_drafts/<skill-name>/SKILL.md

确认后移动到正式目录：
```bash
mv ~/.hermes/skills/_drafts/<skill-name> ~/.hermes/skills/<category>/
```
```

### 6.2 等用户确认

用 clarify 问：确认生成？修改？放弃？

- 确认 → 执行 mv 命令移动到正式目录
- 修改 → 问改什么，回到对应阶段修改
- 放弃 → 保留 _drafts/ 中的文件（不删除）

---

## 反模式

- ❌ 不读目录直接读全文 → 300 页书爆 context
- ❌ 看到书就生成，不判断是否适合 → 浪费时间
- ❌ 编造书中没出现的命令/API → agent 用假命令会翻车
- ❌ 跳过质量验证 → 低质量 skill 进正式目录
- ❌ self-test 走过场，用自己写的 skill 给自己的答案打分 → 必须用书中真实场景，答案是「是或否」而非「好或不好」
- ❌ CRITICAL PRINCIPLES 写成「要全面考虑」这类废话 → 必须可执行

---

## 文件组织

```
~/.hermes/skills/_drafts/<skill-name>/
└── SKILL.md              ← 生成的 skill

确认后 →
~/.hermes/skills/<category>/<skill-name>/
└── SKILL.md
```
